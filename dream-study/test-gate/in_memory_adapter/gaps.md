# In-memory adapter — known gaps

Generated by Phase 0h. The fixture-LLM seam lands ; the broader
adapter-swap mechanism is documented as a forward-looking gap.

## Why a "swap" doesn't exist as a clean facade today

The hecksagon DSL parses `adapter :llm, ...` declarations into IR
(`io_adapters`), but the **runtime path that resolves an adapter kind
to a concrete callable does not yet pivot on test-mode**. As of
2026-05-02 :

- `:llm` adapter — does NOT exist in production. Every Claude call is
  shell-side via `$CLAUDE_BIN` in `body/rem_branch.sh`, `body/musing_mint`,
  etc. (See plan §"Out of scope for this branch", line 932 :
  *":llm adapter — orthogonal but enables full retirement of
  Claude-calling shells."*)
- `:cadence` adapter — does NOT exist in production. Cadence today is
  driven by `hecks-life loop` daemons and the runtime's TickedFeature
  Update path.
- `:daemon` adapter — does NOT exist in production. Daemon spawning is
  done by `hecks-life loop` directly.
- `:fs` adapter — exists for IO but the runtime accesses `.heki` files
  via the Rust `heki` module, not through the Ruby `:fs` adapter
  facade.

So there is no single "adapter resolver" today that we can swap at
test boot. Phase 1 introduces the process_manager primitive ; Phase
1+ (the next branch after dream-study merges) is where `:llm` becomes
a real runtime adapter — and that's the same branch where
`TestAdapters#swap_into` becomes a real implementation rather than a
documented stub.

## Today's seam shape

Tests inject the fixture LLM adapter via **dependency injection at
the call site**. The shape :

```ruby
adapter = Hecks::Adapters::FixtureLlmAdapter.from_yaml(path)
result  = adapter.dream_pulse(cycle: 1, pulse: 1)
```

…which is what `fixture_llm_spec.rb` proves. No hecksagon mutation,
no global monkey-patch, no env var. The PM under test (when it lands
in Phase 4-7) accepts an `:llm` adapter at construction ; its spec
passes the FixtureLlmAdapter ; the real production hecksagon would
pass a future ClaudeLlmAdapter.

## Action items unlocking the full swap

1. **Production `:llm` adapter** — Ruby class with the same call
   surface (`dream_pulse`, `lucid_observation`, etc.) that wraps a
   real Claude HTTP client. Lives at
   `ruby/hecks/adapters/claude_llm_adapter.rb`. (Plan line 917.)
2. **Adapter resolver in runtime** — a `Hecks::Runtime#resolve_adapter(kind)`
   path that consults the loaded hecksagon's IR + a process-level
   registry, returning the concrete adapter instance. Test mode swaps
   the registry, not the bluebook.
3. **`TestAdapters#swap_into` upgrade** — once #1 + #2 exist, this
   method walks the hecksagon's `io_adapters` and registers each test
   counterpart in the registry the runtime consults. The bluebook
   stays unchanged ; the registry is what flips.

## Inbox-worthy items spotted during 0h

- The :llm-via-shell pattern duplicates a lot of bash. `rem_branch.sh`,
  `nrem_branch.sh`, `consolidate.sh`, `mindstream.sh` all carry
  variants of the `[ -x "$CLAUDE_BIN" ] && llm_x=$(claude_call ...)`
  pattern. When the production :llm adapter lands, these all retire
  in one sweep. **The pattern is a single shape duplicated 5 times** —
  same antipattern flagged by the audit for the dispatch-wrapper.
- YAML.safe_load + permitted_classes : Symbol — the fixture file
  intentionally uses string keys and string values throughout. If a
  Phase 4-7 spec wants Symbol keys (e.g. `state: :sleeping` in a
  return value), the adapter will need to expand permitted_classes.
  Worth knowing before someone hits it.
